## 專案核心問題

我想驗證的不是「玩家能不能記住幾個中文字」，而是：

**完全不懂中文的玩家，能不能只靠場景、物件、NPC 行為、重複出現的語句與聲音，自然推測中文詞義？**

因此 prototype 刻意避免：

- 直接翻譯與自動揭曉答案
- 拼音作為預設提示
- 正確／錯誤判定
- 顯式測驗與背單字流程
- 戰鬥、獎勵與複雜成長系統
- 會蓋過語言推測實驗的故事內容

目前的雜貨店不是最終遊戲，而是一個受控制的學習實驗場景。蘋果、茶、水、貨架與店員都有明確的視覺關係，適合觀察玩家能否建立「看到情境、聽到中文、形成猜測」的連結。

## 2026-06-12：建立完整 prototype 架構，也第一次碰到交付管線的成本

### 完成內容

- 建立雜貨店場景、玩家、五名 NPC、三個貨架與對話系統。
- 將詞彙拆成獨立 Godot resources，建立詞彙資料庫與音訊管理架構。
- 加入蘋果、茶、水、角色、地板、牆面與櫃台等素材。
- 建立筆記本的第一版。
- 建立 GitHub Pages Web 部署流程。
- 對齊本機與 CI 的 Godot 4.6.3 版本。
- 修復圖片格式、Godot import、Web export 與舊場景依賴問題。

### 我的想法：場景必須替語言提供證據

我選擇雜貨店，是因為它能用很少的系統提供大量明確線索。玩家不需要任務文字，也能理解「這是貨架」、「這裡放水」、「這個人可能是店員」。如果 NPC 在正確的位置做出合理行為，玩家就有機會把聲音和世界中的物件連起來。

這個選擇也限制了詞彙範圍。prototype 不需要追求單字數量，而需要追求每個詞是否有足夠強的情境。三個清楚的物品可能比二十個沒有行為支撐的詞更有研究價值。

### 我的想法：部署不是附加工作

這一天有大量時間花在 GitHub Pages、Godot 版本、export templates、圖片實際格式與 import cache。這些工作不直接改善學習設計，但它們決定 prototype 能不能被其他人打開。

我原本可能把「能在本機執行」當成完成，但對需要外部測試的作品來說，真正的完成條件是陌生玩家能透過連結進入。部署流程因此不是最後才做的包裝，而是研究工具的一部分。

### 發現與修正

- 副檔名是 `.png` 不代表檔案真的具有正確 PNG 與透明資訊。
- 本機可讀取的資源，不一定能在 Web export 中正確封裝。
- 封存舊程式碼仍可能讓 Godot 掃描到依賴；最後直接移除遺留版本比保留模糊的 Archive 更可靠。
- CI 與本機 Godot 版本不一致，會讓真正的問題被環境差異掩蓋。

### Case study 線索

- 設計決策：使用單一、熟悉、物件關係明確的場所控制實驗範圍。
- 技術教訓：可分享的 prototype 需要從第一週就建立可靠部署。
- 可展示材料：初始 placeholder、素材套用前後、GitHub Pages 失敗與修復流程。

## 2026-06-14：從「有功能」轉向「是否真的支持推測」

這一天的修改最多，也最能反映產品方向是如何透過實際操作逐步收斂的。

### 1. 簡化場景與視覺資訊

#### 完成內容

- 隱藏不必要的 NPC 與貨架標籤。
- 調整鏡頭倍率、角色位置、貨架外觀與移動範圍。
- 簡化部分背景與場景呈現。

#### 我的想法

文字標籤雖然方便開發者辨識節點，對玩家卻可能變成額外答案或視覺干擾。這個 prototype 的重點是玩家從世界本身理解關係，所以場景應該盡可能讓位置、外觀與行為說話。

簡化不代表美術不重要，而是目前每個畫面元素都應該回答一個問題：它有沒有幫助玩家理解情境？如果沒有，它可能只是噪音。

### 2. 重新定義筆記本

#### 迭代過程

- 第一版筆記本曾朝「顯示已解鎖資料」的方向發展。
- 後來加入 encounter／seen tracking，只顯示玩家實際遇過的詞。
- 移除拼音。
- 不再自動顯示英文答案。
- 保留玩家自行輸入英文猜測與重播聲音的能力。
- 筆記本開啟時暫停遊戲，讓玩家可以自主整理理解。

#### 我的想法

這是目前最重要的設計修正之一。

如果筆記本直接顯示中文、拼音與英文，它就會迅速變成傳統單字卡。玩家不再需要觀察世界，只要打開介面取得答案。這會讓 prototype 看起來更「完整」，卻破壞真正想測試的學習假設。

我希望筆記本是一個保存不確定性的地方，而不是消除不確定性的地方。玩家可以記下自己的猜測，稍後再因為新的情境修改它。這種修正過程比系統判定答對更接近我想觀察的自然理解。

#### 尚待驗證

- 玩家是否願意主動打開筆記本？
- 沒有拼音和答案時，玩家會感到好奇還是挫折？
- 玩家會不會在第二次遇到同一詞時修改猜測？
- 重播聲音是否幫助辨識，還是只讓玩家重複點擊而沒有新的情境？

### 3. 讓 NPC 行為成為語意線索，而不是障礙

#### 問題

- 水與茶貨架附近的 NPC 一度不會移動。
- NPC 站位與碰撞會擋住玩家。
- NPC 自動說話會讓對話突然蓋上畫面。
- 玩家與 NPC 對話時仍能移動，削弱了互動的明確性。

#### 修正

- 讓貨架 NPC 在限制區域內持續移動。
- 調整碰撞，避免 NPC 形成不可通過的路障。
- 將 NPC 綁定到合理的貨架與身份位置。
- 改成所有 NPC 都必須由玩家按下互動鍵才開始對話。
- 對話期間凍結玩家。

#### 我的想法

NPC 的移動不是裝飾。當角色在茶或水的貨架附近活動時，他的位置可以幫助玩家推測對話內容；但如果 NPC 把路擋住，玩家注意到的就不再是語言，而是操作上的不順。

同樣地，自動彈出的對話看似增加語言曝光，實際上會奪走玩家的控制權。玩家主動靠近並按下互動鍵，能讓「我現在正在聽這個人說話」成為清楚的事件。凍結移動則避免玩家在語句尚未結束前走掉，也讓聲音、文字與說話者維持一致。

這次迭代讓我更確定：重複不等於強迫曝光。玩家願意主動再次互動，比系統不斷把同一句話推到玩家面前更有價值。

### 4. 從詞彙清單轉向完整語意片段

#### 修改

- 移除「我們」、「你們」、「他們」。
- 移除目前缺少足夠情境證據的「我」、「你」、「他」、「人」、「誰」。
- 加入「這」。
- NPC 對話改為先說「你好」，停頓約三秒，再說「這是 X」。
- 例句包括「你好」與「這是水／茶／蘋果」。
- 筆記本解鎖改為只採用每句對話明確標記的詞彙 id，不再因子字串自動解鎖其他詞。

#### 我的想法

複數代名詞在目前場景中缺少足夠清楚的視覺證據。即使玩家反覆聽到，也很難只靠貨架與單一 NPC 推測出「我們／你們／他們」的差別。

單數代名詞與人物詞也有相同問題。「我」、「你」、「他」、「人」、「誰」雖然常見，但在目前的雜貨店 prototype 裡，玩家缺少足夠清楚的指向、視線與角色動作來驗證猜測。與其讓它們只出現一次並污染筆記本，不如先把它們移出 v0.1 成功指標。

我也決定關閉子字串式自動解鎖。看到「你好」時，玩家遇到的是完整問候語，不代表他已經能把「你」獨立切出來；看到「這是水」時，系統也不應該偷偷把所有子詞都計入。筆記本應該記錄明確設計過的學習接觸，而不是程式字串比對的副產品。

「這」則能直接由 NPC、位置與眼前物品支持。「這是水」是一個完整而且可指向具體物件的語意單位，比孤立顯示「水」更接近真實語言使用。

我也保留「你好」作為固定開場，讓玩家先辨識一個反覆出現的社交片段，再把注意力移到會變化的物品詞。這形成一種自然對照：

- 固定部分：你好／這是
- 變化部分：水、茶、蘋果

這種重複可能幫助玩家自行切分語句，而不需要系統解釋文法。

### Case study 線索

- 關鍵轉折：筆記本由答案資料庫改成玩家猜測空間。
- 關鍵原則：玩家主動互動比自動曝光更符合 agency。
- 行為設計：NPC 的位置、移動與碰撞同時影響語意和可玩性。
- 詞彙策略：刪除缺少情境證據的詞，加入可被指向和比較的完整片段。

## 2026-06-15：開始把 prototype 當成可以測量與維護的研究工具

### 1. 加入經同意的 playtest telemetry

#### 完成內容

- 加入匿名資料分享同意介面。
- 只有玩家明確同意後才記錄事件。
- 記錄互動、遇到的詞、時間與是否輸入筆記本猜測。
- 不傳送玩家輸入的猜測文字、姓名、電子郵件或瀏覽紀錄。
- 透過 Google Apps Script 將資料送往 Google Drive／Sheets 流程。

#### 我的想法

我不想用傳統測驗分數評估這個 prototype，因為那會把研究問題重新拉回「玩家記住多少答案」。更有價值的訊號可能是：

- 玩家是否自願再次與同一物件或 NPC 互動？
- 第一次接觸後多久打開筆記本？
- 是否對某個詞留下猜測？
- 是否在後續接觸後修改猜測？
- 哪些詞有接觸但沒有引發任何後續行動？

同時，學習猜測可能包含玩家的個人表達，所以系統只記錄「有沒有輸入」，而不收集輸入內容。這個限制雖然減少資料細節，但更符合 prototype 階段需要的資料最小化原則。

### 2. 全面檢查互動架構

#### 發現並修正

- NPC 對話泡泡原本離角色過遠，且可能被螢幕上緣裁切。
- 泡泡現在貼近 NPC 上方；靠近頂部時會自動翻到角色下方。
- NPC 對話與貨架對話期間都會凍結玩家。
- 對話中重複按互動鍵不再重複計數或重啟對話。
- 對話時不能打開筆記本。
- 筆記本重播發音不再被誤算為一次新的詞彙接觸。
- 重建筆記本列表不再產生重複卡片。
- 關閉筆記本後會恢復開啟前的暫停狀態。
- Web export 排除測試與舊 build 輸出。

#### 驗證方式

- 建立 `PrototypeSmokeTest.gd`。
- 驗證 11 個詞彙資源與必要場景節點。
- 驗證所有 NPC 的對話與詞彙陣列一致。
- 驗證 NPC 移動與活動範圍。
- 驗證玩家在 NPC／物件對話期間不能移動。
- 驗證泡泡的一般位置與螢幕頂端 fallback。
- 驗證筆記本、音訊重播、暫停狀態與 telemetry consent。
- 執行本機 Web export 與瀏覽器實測。
- 確認 GitHub Pages deployment 成功。

#### 我的想法

當 prototype 開始收集外部測試資料後，互動 bug 就不只是體驗問題，也會污染研究結果。例如玩家重複按互動鍵如果被當成多次學習接觸，最後看到的數據會高估重複行為；筆記本播放聲音若被算成新 encounter，也會混淆「世界中的自然接觸」與「玩家主動複習」。

因此，自動測試不只是工程品質工作。它是在保護研究資料的解釋力。

我也不能宣稱「整個遊戲保證沒有 bug」，但現在至少有一組可重複執行的測試，涵蓋 prototype 最重要的狀態與互動規則。未來每次修改都可以用同一組基準檢查是否破壞核心學習流程。

### Case study 線索

- 研究倫理：先取得同意、資料最小化、不收集猜測內容。
- 數據品質：分清 encounter、重播、重複按鍵與真正的再次互動。
- 工程成熟度：由人工觀察進展到可重複的 smoke test 與部署驗證。
- UI 細節：泡泡位置看似微小，但直接影響說話者與語句的關聯是否清楚。

## 目前的設計原則

### 1. 世界先提供證據，介面不要先給答案

玩家應該先看到物件、角色位置與行動，再接觸中文。英文只能作為內部 metadata 或玩家自己的猜測，不應自動揭曉。

### 2. 教完整語意，不只展示孤立字形

「這是水」比單獨顯示「水」提供更多可推測結構，也讓玩家能透過蘋果、茶、水的替換辨識固定與變化部分。

### 3. 重複要自然，而且最好由玩家主動發起

同一句話自動跳出很多次不等於有效學習。玩家因為好奇再次靠近、互動或重播，才是值得觀察的訊號。

### 4. NPC 行為本身就是教學內容

NPC 的身份、位置、移動範圍、面向與互動對象都應該支持詞義。角色如果站錯位置或阻擋道路，就會同時破壞語意與操作。

### 5. 筆記本保存玩家的推測，不替玩家完成推測

只顯示遇過的中文、允許重播聲音與自行輸入猜測。不顯示拼音、不自動翻譯、不評分。

### 6. 測量探索行為，而不是把 prototype 變成考試

較有價值的資料包括再次互動、猜測建立與修改、停留時間及不同情境間的遷移，而不是答對率。

## 做過但後來修正的方向

這些不是單純的失敗，而是 case study 中最能說明設計判斷的材料。

| 早期方向 | 發現的問題 | 後來的決定 |
| --- | --- | --- |
| 筆記本顯示拼音與英文資訊 | 玩家不需要從情境推測，介面直接成為答案來源 | 移除拼音與自動答案，只保留玩家猜測 |
| 顯示所有詞彙或鎖定項目 | 提前暴露內容，讓玩家把注意力放在收集清單 | 只顯示實際遇過的詞 |
| NPC 自動說話 | 對話打斷探索，玩家缺少主動性 | 所有 NPC 都需按互動鍵 |
| NPC 靜止或具有阻擋碰撞 | 角色失去生活感，並可能堵住動線 | 限制區域內移動，且不阻擋玩家 |
| 使用我／你／他／人／誰／我們／你們／他們 | 場景缺少能清楚區分代名詞與人物詞的證據 | 暫時移除，先集中驗證「你好／這是／水／茶／蘋果」 |
| 子字串自動解鎖詞彙 | 「你好」會順便解鎖「你」，導致筆記本接觸次數失真 | 只記錄每句明確標記的 vocab id |
| 將每次音訊播放視為 encounter | 筆記本重播會污染接觸次數 | 分離世界接觸與自主重播 |
| 只靠人工確認互動 | 修正一處可能讓其他狀態回歸 | 建立涵蓋核心流程的 smoke test |

## 下一輪最值得回答的問題

### 語言推測

1. 完全零基礎玩家第一次聽到「你好，這是水」時，會先推測出哪一部分？
2. 在水、茶、蘋果三個情境後，玩家能否察覺「這是」是固定結構？
3. 玩家會把「你好」理解成問候，還是誤認為物品句的一部分？
4. 文字泡泡和聲音同時出現時，玩家主要依賴哪一種資訊？

### 行為與環境

1. NPC 靠近對應貨架是否足以建立物品關係？
2. NPC 還需要指向、拿取或遞交物品，才能讓詞義更清楚嗎？
3. 玩家是否會主動重複互動，還是只把每個 NPC 點一次？
4. 入口 NPC 是否能自然教會玩家互動方式，而不需要文字教學？

### 筆記本

1. 玩家何時第一次打開筆記本？
2. 沒有拼音與答案的筆記本是否仍被理解為有用工具？
3. 玩家會不會修改原本的猜測？
4. 是否需要保留猜測歷史，讓 case study 能呈現理解如何變化？

### Prototype 範圍

1. 現在的 11 個詞是否太多，還是情境仍足夠清楚？
2. 哪些詞只有文字重複，卻缺乏環境行為支持？
3. 下一個功能應該強化推測證據，還是只增加系統複雜度？

## Case study 建議敘事

### 1. Challenge

傳統語言學習介面通常直接呈現翻譯、拼音或測驗。我想探索：遊戲世界本身能否成為理解陌生語言的媒介？

### 2. Hypothesis

如果中文和清楚的物件、角色身份、位置、動作與自然重複同時出現，即使沒有翻譯，零基礎玩家也可能形成合理猜測。

### 3. Prototype

建立一間包含蘋果、茶、水、店員與顧客的 2D 雜貨店。玩家可自由探索、主動與 NPC／貨架互動，並在無拼音筆記本記錄自己的理解。

### 4. Key iterations

- 筆記本由答案展示改為猜測工具。
- NPC 由自動說話改為玩家主動互動。
- NPC 行為由障礙改為情境線索。
- 詞彙由缺少證據的代名詞改為「這是 X」完整片段。
- telemetry 由一般功能記錄調整為同意制、資料最小化的研究訊號。
- 以自動測試保護互動規則與資料正確性。

### 5. Evidence to collect

- 玩家首次與再次互動的路徑。
- 第一次建立猜測所需時間。
- 猜測是否以及何時被修改。
- 不同詞彙的重複互動率。
- 玩家口述自己使用了哪些線索。
- 玩家誤解的地方，以及造成誤解的場景元素。

### 6. Reflection

這個專案目前最大的學習是：沉浸式設計不是把翻譯藏起來就完成了。當答案被移除後，環境必須承擔更多責任。每個 NPC 的位置、每次互動的時機、每個 UI 選項，都可能決定玩家是在推測語意，還是只是在困惑中點擊。
